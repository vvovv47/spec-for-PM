# 技术可行性分析

## 核心目标

对需求文档中的每个功能单元进行技术可行性分析与判断，确定最优实现技术，从功能需求实现路径、界面与视觉要求满足度、质量与性能要求达标情况三个维度进行全面评估。

## 触发时机

当用户明确要求需要技术可行性分析时调用。

## 核心规则

1. **严格贴合需求场景**：所有技术分析必须基于需求文档中每个功能单元的具体描述，技术的优势必须精准对应需求中的核心目标、约束条件。
2. **不脱离场景的技术科普**：禁止输出与当前功能单元需求无关的技术优点、通用技术介绍。
3. **双输出原则**：技术建议写入需求文档，技术选型理由输出到聊天框。

## 执行步骤

### 第一步：提取功能单元信息

读取 `docs/需求文档/需求文档.md`，提取每个功能单元的以下信息：
- 需求名称
- 触发条件
- 核心功能行为
- 界面与视觉要求
- 质量与性能要求

### 第二步：技术可行性评估

对每个功能单元从以下维度进行评估：
1. **功能需求实现路径**：分析功能的核心逻辑，确定技术实现路线
2. **界面与视觉要求满足度**：评估技术方案能否满足视觉、动画等要求
3. **质量与性能要求达标情况**：评估技术方案能否达到性能、稳定性指标

### 第三步：确定最优技术方案

基于评估结果，确定每个功能单元的最优技术组合：
- 核心技术：实现核心功能所必须的技术
- 辅助技术：辅助实现的技术

### 第四步：输出分析结果

#### 输出形式一：技术建议（写入需求文档）

为 `docs/需求文档/需求文档.md` 中每个功能单元添加「第八点：技术建议」，格式如下：

```
### 8. 技术建议

- **核心技术方案**：[简洁的技术方案说明，一句话说清用什么技术实现什么]
- **辅助技术**：[辅助性技术的简要说明]
```

#### 输出形式二：技术选型理由（输出到聊天框）

每个功能单元输出以下内容：
1. **技术可行性判断**：明确给出「完全可行/有条件可行/不可行」的结论
2. **技术选型分析**：
   - 功能需求实现路径：用什么技术实现这个功能，为什么
   - 界面与视觉满足度：技术方案能否满足视觉要求，如何满足
   - 质量与性能达标情况：技术方案能否达到性能指标
3. **技术边界与风险**：技术方案的天生限制、潜在风险

## 强制输出结构

### 聊天框输出

```
## 技术可行性分析结果

### [功能单元名称]

**技术可行性判断**：完全可行 / 有条件可行 / 不可行

**技术选型理由**：
- 功能需求实现路径：[分析说明]
- 界面与视觉满足度：[分析说明]
- 质量与性能达标情况：[分析说明]

**技术边界与风险**：[简要说明]

---

[下一个功能单元...]
```

### 需求文档更新

在每个功能单元的「质量与性能要求」之后添加：

```
### 8. 技术建议

- **核心技术方案**：[简洁说明]
- **辅助技术**：[简洁说明]
```

## 禁止事项

- 禁止输出代码实现、算法原理、数学公式
- 禁止脱离需求场景进行通用技术科普
- 禁止输出与当前功能单元无关的技术优点
- 禁止不说明理由直接给出技术建议
- 禁止修改功能单元的其他已有内容
