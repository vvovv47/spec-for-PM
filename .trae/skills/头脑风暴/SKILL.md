# 头脑风暴

## 核心目标

补充用户原始产品概念中可能遗漏的关键维度，一次性列出所有可补充点，让用户自主选择需要补充的内容。

## 触发时机

1. 用户首次给出产品概念、需求、功能、效果时
2. 用户对产品的概念、需求、功能、效果进行补充后

## 执行规则

1. **一次性列出所有补充点** — 将6个维度的所有可补充问题一次性列出，不逐个提问
2. **整合用户原始概念** — 先简要复述用户当前的产品概念，确保理解一致
3. **标注补充维度** — 每个补充点标注所属维度，方便用户理解
4. **用户自主选择** — 让用户选择需要补充哪些部分，或提出新的补充点
5. **循环补充** — 当用户补充说明后，再次整合并给出新的补充建议，直到用户满意

## 补充维度清单

### 1. 用户场景维度
- 目标用户是谁？是否有细分用户群体？
- 用户在什么场景下使用？
- 用户的核心痛点是什么？

### 2. 功能完整性维度
- 正常流程是否完整？
- 异常流程如何处理？（网络中断、数据错误、操作超时等）
- 边界情况如何定义？（空数据、最大值、极限场景等）

### 3. 体验维度
- 加载/响应时间要求？
- 是否需要离线可用？
- 错误提示如何呈现？

### 4. 数据维度
- 需要存储哪些数据？
- 数据从哪里来？
- 数据是否需要同步/备份？

### 5. 安全合规维度
- 是否涉及敏感数据？
- 是否需要用户授权/登录？
- 是否符合相关法规要求？

### 6. 运营维度
- 上线后如何收集反馈？
- 是否需要数据埋点统计？
- 后续迭代方向是什么？

## 输出格式

### 第一步：概念整合确认

基于您的描述，我整理出以下产品概念：

> [简要复述用户当前的产品概念，包括核心功能、目标用户、使用场景等]

请确认我的理解是否正确，或需要调整？

### 第二步：可补充维度清单

以下是可能遗漏的维度，您可以：
- 选择需要补充的维度编号
- 提出新的补充点
- 表示"够了，不需要补充"

## 循环补充流程

当用户选择补充某些维度后：
1. 将用户补充的内容整合到产品概念中
2. 基于更新后的概念，重新检查还有哪些维度可以补充
3. 再次列出剩余的可补充点
4. 重复直到用户表示"够了"

## 终止条件

- 用户明确表示"够了"、"不需要补充"、"可以了"
- 用户连续两轮都没有选择新的补充点

## 禁止事项

- 禁止逐个提问补充点，必须一次性列出
- 禁止替用户做选择或假设
- 禁止在用户拒绝后继续追问
