# Spec-for-PM

**规范驱动设计，实现对产品更加精细化的顶层设计。**

本项目是一套面向AI Agent的产品需求文档生成规范体系，通过预定义的Skill（技能）和Rule（规则），帮助AI Agent系统化地生成高质量的需求文档。

---

## 一、项目介绍

### 核心理念

传统的产品需求文档往往存在以下问题：
- 需求描述不完整，遗漏关键细节
- 功能边界模糊，难以指导开发
- 缺少非功能性需求（性能、安全、兼容性等）
- 缺乏从概念到MVP的筛选过程

**Spec-for-PM** 的核心理念是**规范驱动设计**——通过预设的结构化规范，让AI Agent能够系统、全面、层次分明地输出需求文档，确保每个功能单元都包含完整的需求信息。

### 项目目标

- **结构化**：为需求文档定义清晰的结构，每个功能单元都有固定的组成要素
- **完整性**：通过不同的Skill覆盖需求文档的各个方面（功能、交互、视觉、质量、技术选型等）
- **实用性**：支持从完整需求到MVP的筛选，帮助团队聚焦核心价值
- **可扩展**：模块化的Skill设计，便于根据实际需求进行调整和扩展

## 二、快速开始

### 使用方法

由于本项目目前仅包含 `.trae/rules/` 和 `.trae/skills/` 文件，你可以直接将其复制到任何支持Skill机制的项目中使用。

```
your-project/
├── .trae/
│   ├── rules/
│   │   └── project_rules.md    # 项目规则文件
│   └── skills/
│       ├── 头脑风暴/
│       ├── 需求拆解/
│       ├── 技术解释/
│       ├── 技术可行性分析/
│       ├── 其他可行性分析/
│       ├── 交互与业务规则/
│       ├── 界面与视觉/
│       ├── 质量与性能/
│       ├── 全局通用需求/
│       └── MVP化/
└── ...
```

> **注意**：具体的使用方式取决于你使用的AI Agent框架。请查阅对应框架的文档，了解如何加载和触发这些Skill。

## 三、Skill 介绍

以下是项目中所有Skill的功能说明、使用场景和输出内容：

### 1. 头脑风暴

- **使用场景**：当用户给出产品的概念、需求、功能或效果时调用
- **作用**：补充用户可能遗漏的需求点，确保需求完整
- **输出内容**：列出可能需要补充的需求点，供用户选择

### 2. 需求拆解

- **使用场景**：当用户明确提出需要需求拆解时调用
- **作用**：将模糊的产品需求拆解为标准化、无耦合、可落地的模块化功能单元
- **输出内容**：
  - 聊天框：简化确认信息
  - 文件：`docs/需求文档/需求文档.md` 中每个功能单元的完整定义

### 3. 技术解释

- **使用场景**：当用户单独询问某项技术时调用
- **作用**：从业务决策角度解释技术概念，帮助产品经理理解技术选型
- **输出内容**：围绕业务落地决策的技术解释（核心能力、能力边界、选型参考、产品必控项）

### 4. 技术可行性分析

- **使用场景**：当用户明确要求需要技术可行性分析时调用
- **作用**：对每个功能单元进行技术可行性评估，确定最优实现技术
- **输出内容**：
  - 聊天框：详细的技术选型理由（功能实现路径、视觉满足度、性能达标情况、技术边界与风险）
  - 文件：`docs/需求文档/需求文档.md` 中每个功能单元的「第八点：技术建议」

### 5. 其他可行性分析

- **使用场景**：当用户明确需要可行性分析时，在技术可行性分析之后调用
- **作用**：分析业务、市场、用户体验等方面的可行性
- **输出内容**：非技术层面的可行性分析报告

### 6. 交互与业务规则

- **使用场景**：当用户明确需要交互与业务逻辑时调用
- **作用**：为功能单元补充交互流程、异常处理、专属业务规则
- **输出内容**：
  - 聊天框：简化确认信息
  - 文件：`docs/需求文档/需求文档.md` 中每个功能单元的「第五点：交互与业务规则」

### 7. 界面与视觉

- **使用场景**：当用户明确需要界面与视觉时调用
- **作用**：用纯中文自然语言描述用户可见的视觉效果和风格感受
- **输出内容**：
  - 聊天框：简化确认信息
  - 文件：`docs/需求文档/需求文档.md` 中每个功能单元的「第六点：界面与视觉要求」

### 8. 质量与性能

- **使用场景**：当用户明确需要质量与性能设计时调用
- **作用**：为功能单元补充可量化、可验收的质量标准
- **输出内容**：
  - 聊天框：简化确认信息
  - 文件：`docs/需求文档/需求文档.md` 中每个功能单元的「第七点：质量与性能要求」

### 9. 全局通用需求

- **使用场景**：当用户明确需要全局通用需求时调用
- **作用**：生成需求文档第二部分的全局通用模块内容
- **输出内容**：
  - 聊天框：简化确认信息
  - 文件：`docs/需求文档/需求文档.md` 开头的全局通用模块

### 10. MVP化

- **使用场景**：当用户明确需要MVP版需求文档时调用
- **作用**：基于完整需求文档，筛选最小可行功能集合
- **输出内容**：
  - 聊天框：MVP筛选说明
  - 文件：`docs/需求文档/MVP需求文档.md`

## 五、已知问题

### 1. 输出精炼度

本项目处于初级阶段，当前输出的需求内容可能存在以下问题：
- 内容较为臃肿，不够精炼
- 部分描述不够准确或到位
- 结构可能过于复杂

### 2. Agent兼容性问题

不同的AI Agent和大模型对Skill的遵守程度可能不同：
- 某些Agent可能不完全遵循Skill中定义的输出格式
- 大模型的指令遵循能力差异可能导致输出质量不一致
- 部分Agent可能跳过某些Skill或改变输出顺序

### 3. 持续迭代

本项目需要在更多的实际项目中进行实践和调整：
- 收集使用反馈，持续优化各Skill的输出质量
- 根据不同产品类型调整需求文档的结构
- 补充更多针对特定场景的Skill
